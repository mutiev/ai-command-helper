# Testing

`ai` не имеет автоматизированных тестов — инструмент взаимодействует с внешним API
и файловой системой, что делает unit-тесты избыточными для его масштаба.

Этот документ описывает **ручное тестирование**: как проверить каждый компонент
после изменений, какие граничные случаи важны и как изолировать API от локальной логики.

---

## Быстрая проверка после изменений

Минимальный набор команд чтобы убедиться что ничего не сломано:

```bash
# 1. Синтаксис Python
python3 -m py_compile ~/.local/bin/ai && echo "OK: синтаксис"

# 2. Справка работает (не требует API)
ai --help

# 3. Один запрос к API
ai "напиши echo hello"

# 4. Контекст каталога
ai "сколько файлов в текущем каталоге?"

# 5. Интерактивный режим — ввести вопрос, затем exit
ai
```

Если все пять пунктов прошли — базовый smoke test пройден.

## Компоненты и как их тестировать

### 1. Разбор аргументов CLI

Не требует API. Проверяется флагом `--help` и намеренно некорректными аргументами.

```bash
# Справка
ai --help

# Неизвестный флаг — должен вывести ошибку argparse и exit 2
ai --unknown-flag 2>&1; echo "exit: $?"

# Пустой вопрос (пробел) — должен войти в интерактивный режим
ai " "
```

### 2. Разрешение API-ключа

Проверяем все три уровня waterfall.

```bash
# Уровень 1: переменная окружения имеет приоритет над файлом
ANTHROPIC_API_KEY="sk-ant-test-from-env" ai "привет" 2>&1 | head -5
# Ожидание: запрос уйдёт с ключом из env, а не из файла
# (можно проверить через tcpdump или временно подменить файл)

# Уровень 2: ключ из файла
cat ~/.config/ai/api_key   # должен существовать и быть непустым
ls -la ~/.config/ai/api_key  # права должны быть 600

# Уровень 3: отсутствие ключа — интерактивный запрос
mv ~/.config/ai/api_key ~/.config/ai/api_key.bak
unset ANTHROPIC_API_KEY
ai "тест"
# Ожидание: показывает UI запроса ключа через getpass
# Ввести валидный ключ → должен сохранить и продолжить
mv ~/.config/ai/api_key.bak ~/.config/ai/api_key

# Уровень 3: отсутствие ключа + stdin не TTY (pipe-режим)
mv ~/.config/ai/api_key ~/.config/ai/api_key.bak
echo "вопрос" | ai
# Ожидание: сообщение "stdin не является терминалом" + exit 1
echo "exit: $?"
mv ~/.config/ai/api_key.bak ~/.config/ai/api_key
```

**Проверка валидации формата ключа:**

```bash
mv ~/.config/ai/api_key ~/.config/ai/api_key.bak
unset ANTHROPIC_API_KEY
# Запустить ai, ввести невалидный ключ "hello"
# Ожидание: "Ключ должен начинаться с sk-ant-" + exit 1
ai "тест"
mv ~/.config/ai/api_key.bak ~/.config/ai/api_key
```

### 3. Сборка контекста — `build_context()`

Проверяем что контекст корректно формируется и передаётся модели.

```bash
# Базовый: модель видит файлы текущего каталога
cd /tmp && touch test_file_alpha.txt test_file_beta.txt
ai "перечисли файлы которые ты видишь"
# Ожидание: модель называет test_file_alpha.txt и test_file_beta.txt

# Флаг -d: дополнительный каталог
ai -d /etc "что за файлы в /etc?"
# Ожидание: модель описывает содержимое /etc

# Флаг -f: конкретный файл
echo "PORT=8080" > /tmp/test_config.env
ai -f /tmp/test_config.env "какой порт в конфиге?"
# Ожидание: модель называет 8080

# Несколько флагов одновременно
ai -d /tmp -f /etc/hostname "как называется хост и что в /tmp?"

# Несуществующий каталог в -d — не должен падать
ai -d /nonexistent_dir_xyz "привет"
# Ожидание: ai запускается, модель получает "Каталог не существует: /nonexistent_dir_xyz"

# Несуществующий файл в -f
ai -f /nonexistent_file.txt "привет"
# Ожидание: аналогично, "Файл не существует: /nonexistent_file.txt"

# Системная информация: hostname и uname в контексте
ai "как называется эта машина и какая ОС?"
# Ожидание: модель называет hostname и версию ядра без использования инструментов
```

### 4. Инструменты — `list_directory`, `read_file`, `run_command`

#### `list_directory`

```bash
# Базовый
ai "покажи содержимое /tmp"
# Ожидание: вывод → list_directory({"path": "/tmp"})

# Каталог без доступа
ai "что в /root?"
# Ожидание: модель получает "Нет доступа к каталогу: /root"

# Несуществующий каталог
ai "что в /zzz_not_exist?"
# Ожидание: "Каталог не существует: /zzz_not_exist"

# Пустой каталог
mkdir /tmp/empty_test_dir
ai "что в /tmp/empty_test_dir?"
# Ожидание: "(каталог пуст)"
rmdir /tmp/empty_test_dir

# Лимит 200 записей: каталог с >200 файлами
mkdir /tmp/bigdir_test && for i in $(seq 1 250); do touch /tmp/bigdir_test/f$i; done
ai "сколько файлов в /tmp/bigdir_test?"
# Ожидание: модель видит "... и ещё 50 объектов" и говорит о 200+
rm -rf /tmp/bigdir_test
```

#### `read_file`

```bash
# Базовый
echo "Hello, world!" > /tmp/test_read.txt
ai "что написано в /tmp/test_read.txt?"
# Ожидание: вызов read_file, ответ "Hello, world!"

# Файл без доступа
sudo touch /tmp/no_access_test.txt
sudo chmod 000 /tmp/no_access_test.txt
ai "прочитай /tmp/no_access_test.txt"
# Ожидание: "Нет доступа к файлу"
sudo rm /tmp/no_access_test.txt

# Бинарный файл
ai "прочитай /bin/ls"
# Ожидание: "[бинарный файл, N байт] — текстовое содержимое недоступно"

# Файл > 32 КБ
dd if=/dev/urandom bs=1 count=40000 2>/dev/null | base64 > /tmp/bigfile_test.txt
ai "прочитай первые строки /tmp/bigfile_test.txt"
# Ожидание: модель читает файл, в конце ответа "[обрезано, показано 32768 из ... байт]"
rm /tmp/bigfile_test.txt

# UTF-8 с кириллицей
echo "Привет мир" > /tmp/test_utf8.txt
ai "что в /tmp/test_utf8.txt?"
# Ожидание: "Привет мир"

# Пустой файл
touch /tmp/empty_test.txt
ai "что в /tmp/empty_test.txt?"
# Ожидание: корректная обработка, модель сообщает что файл пустой
rm /tmp/empty_test.txt /tmp/test_read.txt /tmp/test_utf8.txt
```

#### `run_command`

```bash
# Базовый — должен показать ⚙ в stderr
ai "сколько места на диске?" 2>&1
# Ожидание: строка "  ⚙ df -h" в stderr, затем ответ

# Таймаут
ai "запусти sleep 30"
# Ожидание: через 10с "[таймаут 10с]", модель сообщает о таймауте

# Несуществующая команда
ai "запусти команду zzz_no_such_cmd_xyz"
# Ожидание: stderr с "command not found", код возврата != 0

# Пайп в команде
ai "найди все python процессы"
# Ожидание: модель вызывает что-то вроде "ps aux | grep python" — пайп работает

# Команда с ненулевым кодом возврата
ai "проверь существует ли файл /zzz_not_exist"
# Ожидание: модель получает "[код возврата: 1]" и корректно интерпретирует
```

### 5. Agentic loop — `ask()`

Проверяем что цикл tool use завершается корректно.

```bash
# Один вызов инструмента
ai "что за файлы в /tmp?"
# Ожидание: один вызов list_directory → финальный ответ

# Цепочка инструментов: сначала найти файл, потом прочитать
echo "DB_HOST=localhost" > /tmp/chain_test.env
ai "найди .env файл в /tmp и покажи его содержимое"
# Ожидание: list_directory("/tmp") → read_file("/tmp/chain_test.env") → ответ
rm /tmp/chain_test.env

# Глубокая цепочка: несколько каталогов
ai "сравни что есть в /tmp и /var/tmp"
# Ожидание: два вызова list_directory → сравнительный ответ
```

### 6. Рендер вывода — `render()`

```bash
# Bash-блоки должны подсвечиваться в TTY
ai "покажи команду для просмотра открытых портов"
# Ожидание: блок с ╭─ bash ... ╰───── выделен цветом

# В pipe — без ANSI-кодов
ai "покажи команду для просмотра открытых портов" | cat
# Ожидание: чистый текст, никаких \033[... escape-последовательностей

# Вывод в файл
ai "напиши команду" > /tmp/ai_output.txt
cat /tmp/ai_output.txt
# Ожидание: читаемый текст без мусора

# Несколько блоков кода в одном ответе
ai "покажи 3 разных способа найти большие файлы"
# Ожидание: три отдельных ╭─ bash ╰───── блока
rm /tmp/ai_output.txt
```

### 7. Интерактивный режим

```bash
# Базовый запуск
ai
# Ввести: "как дела?"  → получить ответ
# Ввести: "что ты помнишь из предыдущего вопроса?"
# Ожидание: модель помнит контекст диалога

# Команды выхода
ai
# Ввести: exit   → выход
ai
# Ввести: quit   → выход
ai
# Ввести: q      → выход
ai
# Ввести: выход  → выход

# Ctrl+C → выход без ошибки
ai
# Нажать Ctrl+C
# Ожидание: чистый выход, без Python traceback

# EOF (Ctrl+D) → выход без ошибки
ai
# Нажать Ctrl+D
# Ожидание: чистый выход

# Кириллица в интерактивном режиме — ключевой edge case
ai
# Ввести: "как найти большие файлы?"
# Ожидание: ответ без UnicodeDecodeError
```

### 8. Кастомный системный промпт

```bash
# Создать кастомный промпт
cat > ~/.config/ai/system_prompt << 'EOF'
Отвечай только одним словом: "ДА" или "НЕТ".
EOF

ai "земля круглая?"
# Ожидание: "ДА"

ai "2 + 2 = 5?"
# Ожидание: "НЕТ"

# Удалить и проверить что вернулся дефолт
rm ~/.config/ai/system_prompt
ai "что ты умеешь?"
# Ожидание: стандартный ответ про терминальный ассистент
```

### 9. Установщик `install.sh`

```bash
# Синтаксис bash
bash -n install.sh && echo "OK: синтаксис"

# Shellcheck (если установлен)
shellcheck install.sh

# Тест переменной AI_INSTALL_DIR
mkdir /tmp/test_install_dir
AI_INSTALL_DIR=/tmp/test_install_dir bash install.sh
ls -la /tmp/test_install_dir/ai
# Ожидание: файл ai существует и исполняемый
rm -rf /tmp/test_install_dir

# Тест pipe-режима (имитация curl | bash)
bash install.sh < /dev/null
# Ожидание: предупреждение "ai запросит ключ при первом запуске", exit 0

# Idempotency: повторная установка не ломает существующую
bash install.sh
bash install.sh
ai --help
# Ожидание: работает после двух установок подряд
```

## Граничные случаи — сводная таблица

| Сценарий | Компонент | Ожидаемое поведение |
|---|---|---|
| `ANTHROPIC_API_KEY` задан и файл тоже есть | `read_api_key` | Используется env, файл игнорируется |
| Файл `api_key` пустой | `read_api_key` | Переходит к интерактивному запросу |
| Ключ не начинается с `sk-ant-` | `read_api_key` | Отказ с сообщением, exit 1 |
| stdin не TTY и нет ключа | `read_api_key` | Инструкция по ручной настройке, exit 1 |
| `-d` несуществующий путь | `build_context` | Запускается, модель видит сообщение об ошибке |
| `-f` бинарный файл | `read_file_safe` | Сообщение о бинарном файле, не краш |
| Файл > 32 КБ | `read_file_safe` | Обрезка с явным уведомлением |
| Каталог > 200 файлов | `list_dir` | Первые 200 + "... и ещё N объектов" |
| `run_command` зависает | `run_shell` | Таймаут 10с, возврат строки с описанием |
| Ответ API без tool_use | `ask` | Возвращает текст, цикл завершается |
| Ctrl+C в интерактиве | `main` | Чистый выход, без traceback |
| Ctrl+D в интерактиве | `main` | Чистый выход (пустая строка из buffer.readline) |
| ANSI в pipe/redirect | `render`, `color` | Нет escape-кодов, чистый текст |
| Кириллица на вводе | интерактивный цикл | Корректное декодирование UTF-8 |
| `exit` / `quit` / `q` / `выход` | интерактивный цикл | Выход без ошибки |
| Пустой ввод в интерактиве | интерактивный цикл | Выход (readline вернул пустую строку) |
| Повторная установка | `install.sh` | Перезаписывает файл, не дублирует PATH |

## Изоляция от реального API при разработке

Если нужно тестировать логику без отправки запросов в Anthropic:

```bash
# Подменить функцию ask() временным stub — добавить в конец ai.py перед if __name__:
# def ask(client, messages, system):
#     return "```bash\necho 'stub response'\n```"
```

Или использовать невалидный ключ и проверять только до момента API-вызова:

```bash
# Проверить что build_context работает корректно (упадёт на API но контекст уже собран)
ANTHROPIC_API_KEY="sk-ant-invalid" ai "тест" 2>&1
# Ожидание: ошибка аутентификации от API, но не Python traceback из build_context
```

## Что добавить при росте кодовой базы

Когда файл вырастет за ~1000 строк или появится несколько контрибьюторов, стоит добавить:

- `pytest` с моком `anthropic.Anthropic` для тестирования `ask()` и `dispatch_tool()`
- `pytest-tmp-path` для изолированного тестирования `list_dir()` и `read_file_safe()`
- `shellcheck` в CI для `install.sh`
- smoke-тест в GitHub Actions: установка + `ai --help` на Ubuntu и Debian
