# Architecture

## Обзор

`ai` — однофайловый CLI-ассистент для работы в терминале Linux/SSH.
Принципиальное архитектурное решение: **один Python-скрипт, две внешние зависимости (`anthropic`, `questionary`)**.
Никакого сервера, демона, базы данных или конфигурационного фреймворка.

```
пользователь
    │
    ▼
  ai.py                         ~/.config/ai/
┌─────────────────────┐        ┌──────────────────┐
│  CLI (argparse)     │◄──────►│  api_key         │
│  interactive_menu() │        │  system_prompt   │
│  build_context()    │        │  sessions/       │
│  ask()  ──loop──►  │        └──────────────────┘
│    Anthropic API    │
│  ◄── tool_use ──   │        файловая система
│  dispatch_tool()   │◄──────► (cwd, -d, -f флаги)
│  render()          │
└─────────────────────┘
    │
    ▼
  stdout (терминал)
```

## Компоненты

### 1. Точка входа — `main()`

Разбирает аргументы и маршрутизирует выполнение в один из трёх режимов:

| Режим | Условие | Поведение |
|---|---|---|
| `--setup` | флаг `--setup` | только сохраняет API-ключ, выходит |
| `--sessions` | флаг `--sessions` | показывает список сессий, выходит |
| `--clear-session` | флаг `--clear-session ID` | удаляет сессию, выходит |
| Одиночный запрос | передан `query` аргумент | один вопрос → ответ → сохранение сессии → выход |
| Интерактивный | нет `query` | TUI-меню → REPL-цикл с автосохранением |

### 2. Сборка контекста — `build_context()`

Вызывается один раз перед стартом, формирует **системный промпт** из нескольких слоёв:

```
DEFAULT_SYSTEM (или ~/.config/ai/system_prompt)
    +
## Рабочий каталог: /current/path
[листинг cwd]
    +
## Каталог: /path  (для каждого -d флага)
[листинг]
    +
## Файл: path  (для каждого -f флага)
[содержимое файла]
    +
## Система: hostname / uname
```

Контекст передаётся как часть `system` параметра API, а не как `user` сообщение — это гарантирует что он всегда присутствует в диалоге и не вытесняется из истории.

### 3. Цикл запросов — `ask()`

Реализует **agentic loop** — бесконечный цикл обращений к API до получения финального ответа без tool_use:

```
messages → API → response
                    │
          ┌─────────┴──────────┐
          │ stop_reason?       │
          │                    │
     tool_use              end_turn
          │                    │
   dispatch_tool()        return text
          │
   tool_result → messages
          │
         (повтор)
```

Вся история сообщений (`messages`) хранится в памяти текущего процесса и автоматически сохраняется на диск после каждого обмена (см. раздел "6. Хранилище сессий").

### 4. Инструменты — `TOOLS` + `dispatch_tool()`

Три инструмента, доступных модели:

```
list_directory(path)
    └─► list_dir()
            └─► Path.iterdir() + stat()
                Лимит: 200 записей
                Сортировка: каталоги первыми, затем файлы по имени

read_file(path)
    └─► read_file_safe()
            └─► open("rb") → decode("utf-8", errors="replace")
                Лимит: 32 768 байт (32 КБ)
                Детектор бинарных файлов: наличие байта 0x00

run_command(command)
    └─► run_shell()
            └─► subprocess.run(shell=True, capture_output=True)
                Таймаут: 10 секунд
                CWD: текущий рабочий каталог пользователя
```

Все три инструмента работают с правами текущего пользователя — никакого повышения привилегий.

### 5. Управление API-ключом — `read_api_key()`

Трёхуровневый waterfall:

```
1. os.environ["ANTHROPIC_API_KEY"]   ← наивысший приоритет
        ↓ не найден
2. ~/.config/ai/api_key              ← персистентное хранилище
        ↓ не найден
3. getpass() → сохранить в файл      ← интерактивный onboarding
        ↓ stdin не tty
   вывод инструкции → sys.exit(1)
```

Файл ключа создаётся с правами `600` (только владелец).

### 6. Хранилище сессий

Сессии привязаны к рабочему каталогу (cwd). Ключ каталога — `sha256(cwd)[:8]`.

```
~/.config/ai/sessions/
└── a1b2c3d4/               # sha256("/home/user/project")[:8]
    ├── _meta.json           # {"cwd": "/home/user/project"}
    ├── deploy.json          # именованная сессия (-s deploy)
    └── 2026-03-30T14:22:05.json  # автоматическая сессия
```

Формат JSON-файла сессии:

```json
{
  "started":  "2026-03-30T14:22:05",
  "updated":  "2026-03-30T14:25:12",
  "preview":  "почему nginx не стартует?",
  "messages": [
    {"role": "user",      "content": "..."},
    {"role": "assistant", "content": "..."}
  ]
}
```

Функции:
- `save_session(id, messages)` — сериализует ContentBlock через `model_dump()`, записывает JSON
- `load_session(id)` — загружает messages из файла текущего cwd
- `list_sessions(cwd_only)` — список сессий для текущего каталога или всех
- `delete_session(id)` — удаляет JSON-файл

Автосохранение: в интерактивном режиме сессия сохраняется после каждого обмена. В одиночном режиме — один раз после ответа (если не `--no-save`).

### 7. Интерактивное меню — `interactive_menu()`

При запуске `ai` без аргументов (TTY-режим), `questionary` отображает два экрана:

1. **Выбор сессии** (`questionary.select`) — «Новая сессия» или одна из существующих для текущего cwd
2. **Выбор контекста** (`questionary.checkbox`) — файлы и каталоги из cwd (top-10 по `_top_entries()`)

`_top_entries()` возвращает до 10 видимых (не dotfiles) не-бинарных записей из cwd, отсортированных: каталоги первыми, затем по mtime (свежие выше).

### 8. Рендеринг — `render()`

Построчный парсер Markdown-подобного вывода. Обнаруживает блоки ```` ``` ```` и подсвечивает их зелёным+жирным. Остальной текст — без изменений.

Если stdout не является TTY (пайп, редирект) — ANSI-коды не добавляются, текст выводится как есть.

## Поток данных для одного запроса

```
$ ai -d /etc/nginx "почему nginx не стартует?"
        │
        ▼
   argparse → args.dir=["/etc/nginx"], args.query="почему nginx не стартует?"
        │
        ▼
   read_api_key()  →  "sk-ant-..."
        │
        ▼
   build_context(["/etc/nginx"], [])
        │  list_dir("/current/cwd")
        │  list_dir("/etc/nginx")
        │  run_shell("hostname"), run_shell("uname -sr")
        ▼
   full_system = DEFAULT_SYSTEM + контекст
        │
        ▼
   messages = [{"role": "user", "content": "почему nginx не стартует?"}]
        │
        ▼
   ask(client, messages, full_system)
        │
        │  → API call #1
        │  ← tool_use: read_file("/etc/nginx/nginx.conf")
        │  → dispatch_tool() → read_file_safe(...)
        │  → API call #2
        │  ← end_turn: "В строке 42 синтаксическая ошибка..."
        ▼
   render(answer)  →  stdout
```

## Конфигурационные файлы

```
~/.config/ai/
├── api_key          # Anthropic API ключ, chmod 600
├── system_prompt    # (опционально) замена DEFAULT_SYSTEM целиком
└── sessions/        # Персистентные сессии
    └── {cwd_hash}/  # SHA256[:8] от полного пути cwd
        ├── _meta.json   # {"cwd": "/full/path"}
        └── {id}.json    # сессия: started, updated, preview, messages
```

Нет поддержки частичного переопределения — `system_prompt` заменяет весь дефолтный промпт.

## Ограничения текущей архитектуры

- `system_prompt` заменяется целиком, нет механизма “append”
- `run_command` выполняется с `shell=True` — потенциальный риск при инъекции через имена файлов
- Нет retry-логики при ошибках API (rate limit, 5xx)
- Сессии не имеют автоочистки — удаление через `--clear-session` вручную
- Нет стриминга ответа — пользователь ждёт весь ответ целиком
- Контекст (`-d`, `-f`) собирается один раз при запуске, не обновляется в процессе диалога
