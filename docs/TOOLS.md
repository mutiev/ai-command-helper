# Tools Reference

Документация по инструментам (tool use), доступным модели Claude во время работы `ai`.

Инструменты объявлены в массиве `TOOLS` в `ai.py` и диспетчеризируются через `dispatch_tool()`.
Модель вызывает их **автоматически** — пользователь видит только строки вида `→ tool_name(...)` в stderr.

---

## Общие принципы

- Все инструменты выполняются с правами **текущего Unix-пользователя** — без эскалации привилегий
- Ни один инструмент не изменяет файловую систему или состояние системы
- Ошибки (нет файла, нет доступа, таймаут) не бросают исключений — возвращают строку с описанием проблемы, которую модель читает и учитывает
- Вызовы инструментов логируются в **stderr** (не stdout), чтобы не мешать пайпам

## `list_directory`

Возвращает содержимое каталога файловой системы.

### Сигнатура (в terms API)

```json
{
  "name": "list_directory",
  "input": {
    "path": "/absolute/or/relative/path"
  }
}
```

### Реализация

```
dispatch_tool("list_directory", {"path": "..."})
    └─► list_dir(path, max_entries=200)
            └─► Path(path).iterdir()
```

### Формат вывода

```
d            .git
d            src
d            tests
-      1 234  README.md
-     45 678  ai.py
-      3 210  install.sh
```

Колонки: `тип` · `размер в байтах (только для файлов)` · `имя`.

Тип: `d` — каталог, `-` — файл, `?` — другое (симлинк, сокет и т.п.).

Сортировка: каталоги первыми, затем файлы — оба блока по алфавиту.

### Ограничения

| Параметр | Значение | Причина |
|---|---|---|
| Максимум записей | 200 | Предотвращает переполнение контекста при листинге `/usr/bin` |
| Рекурсия | нет | Каждый уровень — отдельный вызов инструмента |

При превышении лимита последней строкой добавляется:
```
... и ещё 1547 объектов
```

### Граничные случаи

| Ситуация | Возвращает |
|---|---|
| Путь не существует | `"Каталог не существует: /path"` |
| Путь — файл, не каталог | `"Не каталог: /path"` |
| Нет прав на каталог | `"Нет доступа к каталогу: /path"` |
| Нет прав на отдельный элемент | строка с `[нет доступа]` для конкретного файла |
| Пустой каталог | `"(каталог пуст)"` |

### Когда модель его вызывает

- При вопросах о структуре проекта: _"что за файлы здесь?"_
- Для навигации перед `read_file`: найти нужный файл по имени
- При вопросах типа _"есть ли тут конфиг nginx?"_
- При флаге `-d /path` — контекст уже включён в system-промпт, но модель может запросить подкаталоги

## `read_file`

Читает содержимое текстового файла.

### Сигнатура

```json
{
  "name": "read_file",
  "input": {
    "path": "/path/to/file"
  }
}
```

### Реализация

```
dispatch_tool("read_file", {"path": "..."})
    └─► read_file_safe(path, max_bytes=32_768)
            └─► open(path, "rb")
                    └─► read(32_768)
                    └─► detect binary (0x00 byte)
                    └─► decode("utf-8", errors="replace")
```

### Ограничения

| Параметр | Значение | Причина |
|---|---|---|
| Максимум байт | 32 768 (32 КБ) | ≈8 000 токенов — значимая часть контекста, но не весь |
| Кодировка | UTF-8 | Стандарт для конфигов и скриптов на Linux |
| Ошибки декодирования | `errors="replace"` | Некорректные байты → `?`, не краш |

При обрезке в конец файла добавляется:
```
... [обрезано, показано 32768 из 158432 байт]
```

### Детектор бинарных файлов

Перед декодированием проверяется наличие нулевого байта (`0x00`) в прочитанном блоке.
Если найден — файл считается бинарным:
```
[бинарный файл, 2048576 байт] — текстовое содержимое недоступно
```

Это предотвращает передачу бинарного мусора в контекст.

**Известное ограничение:** `0x00` может встречаться в некоторых текстовых форматах (UTF-16, старые Windows-файлы). Такие файлы будут ошибочно определены как бинарные.

### Граничные случаи

| Ситуация | Возвращает |
|---|---|
| Файл не существует | `"Файл не существует: /path"` |
| Путь — каталог | `"Не файл: /path"` |
| Нет прав | `"Нет доступа к файлу: /path"` |
| Бинарный файл | `"[бинарный файл, N байт] — ..."` |
| Файл > 32 КБ | содержимое + сообщение об обрезке |
| Пустой файл | пустая строка `""` |

### Когда модель его вызывает

- Анализ конфигурационных файлов: `/etc/nginx/nginx.conf`, `docker-compose.yml`
- Чтение логов: `/var/log/syslog`, `error.log`
- Просмотр скриптов и кода
- При флаге `-f /path` — файл уже передан через system-промпт, но модель может запросить другие файлы по ходу диалога

## `run_command`

Выполняет shell-команду и возвращает её вывод.

### Сигнатура

```json
{
  "name": "run_command",
  "input": {
    "command": "df -h"
  }
}
```

### Реализация

```
dispatch_tool("run_command", {"command": "..."})
    └─► print(f"  ⚙ {cmd}", stderr)     ← логируется пользователю
    └─► run_shell(cmd, timeout=10)
            └─► subprocess.run(
                    cmd,
                    shell=True,
                    capture_output=True,
                    text=True,
                    timeout=10,
                    cwd=os.getcwd()      ← CWD пользователя
                )
```

### Ограничения

| Параметр | Значение | Причина |
|---|---|---|
| Таймаут | 10 секунд | Предотвращает зависание на `tail -f`, `watch`, интерактивных командах |
| `shell=True` | да | Необходим для пайпов: `ps aux \| grep nginx` |
| CWD | текущий каталог пользователя | Относительные пути в командах работают ожидаемо |

### Формат вывода

Возвращается объединённая строка из:
```
[stdout]
[stderr]:        ← префикс, только если stderr непустой
[код возврата: N]  ← только если код != 0
```

### Граничные случаи

| Ситуация | Возвращает |
|---|---|
| Команда ничего не выводит | `"(нет вывода)"` |
| Таймаут | `"[таймаут 10с]"` |
| Команда не найдена | stderr с `command not found` + код возврата |
| Любое исключение Python | `"[ошибка: {e}]"` |

### Видимость для пользователя

Каждый вызов `run_command` немедленно выводит команду в **stderr**:
```
  ⚙ df -h
```

Это единственный инструмент, вызов которого явно показывается пользователю в реальном времени
(остальные логируются через `→ tool_name(...)`, но только в verbose-режиме, который пока не реализован).

### Назначение vs ограничения

**Предназначен для:**
```bash
df -h                          # свободное место
ps aux | grep nginx            # процессы
uname -a                       # версия ядра
systemctl status myservice     # статус сервиса
free -h                        # память
netstat -tlnp                  # открытые порты
journalctl -u nginx --since "1 hour ago"  # логи systemd
find /etc -name "*.conf" -type f          # поиск файлов
```

**Не предназначен для (но технически возможно):**
```bash
rm -rf /                       # деструктивные операции
systemctl restart nginx        # изменение состояния сервисов
apt install something          # установка пакетов
```

Граница соблюдается через описание инструмента в промпте:
> "Используй только для чтения данных. НЕ используй для команд, изменяющих систему."

Это мягкое ограничение — модель следует ему, но архитектура не блокирует деструктивные команды технически. Реальная граница безопасности — права Unix-пользователя (см. ADR-005 в DECISIONS.md).

## Диспетчер — `dispatch_tool()`

Центральная точка маршрутизации вызовов инструментов:

```python
def dispatch_tool(name: str, inp: dict) -> str:
    if name == "list_directory":
        return list_dir(inp.get("path", "."))
    elif name == "read_file":
        return read_file_safe(inp.get("path", ""))
    elif name == "run_command":
        cmd = inp.get("command", "")
        print(color(f"  ⚙ {cmd}", DIM), file=sys.stderr)
        return run_shell(cmd)
    return f"Неизвестный инструмент: {name}"
```

Все инструменты возвращают `str`. Результат оборачивается в `tool_result` и отправляется обратно в API.

## Как добавить новый инструмент

1. Добавить описание в массив `TOOLS`:

```python
{
    "name": "my_tool",
    "description": "Что делает инструмент — модель читает это при выборе.",
    "input_schema": {
        "type": "object",
        "properties": {
            "param": {
                "type": "string",
                "description": "Описание параметра",
            }
        },
        "required": ["param"],
    },
},
```

2. Добавить ветку в `dispatch_tool()`:

```python
elif name == "my_tool":
    return my_tool_function(inp.get("param", ""))
```

3. Реализовать функцию — она должна:
   - Принимать строковые аргументы
   - Возвращать `str` (модель получает только текст)
   - Никогда не бросать исключений наружу — ловить их и возвращать описание ошибки
   - Не изменять состояние системы (или явно задокументировать что делает)

4. Добавить запись в этот файл.
