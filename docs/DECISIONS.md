# Architecture Decision Records

Журнал архитектурных решений проекта `ai`.
Каждая запись объясняет **что** было решено, **почему** и какие **альтернативы** отвергнуты.

---

## ADR-001: Один файл вместо пакета

**Статус:** принято

**Контекст:**
Инструмент устанавливается на множество SSH-машин с разными окружениями.
Цель — минимальное трение при установке и обновлении.

**Решение:**
Весь код живёт в одном файле `ai.py`. Никаких `__init__.py`, пакетов, точек входа через setuptools.

**Причины:**
- Установка = один `curl` + `chmod +x`. Никакого `pip install .` с правами или venv.
- Обновление = перезаписать один файл. Никаких конфликтов версий.
- Читаемость для случайного `cat ~/.local/bin/ai` — весь код виден сразу.
- На SSH-машинах часто нет прав на глобальную установку пакетов.

**Отвергнутые альтернативы:**
- `pip`-пакет с `pyproject.toml` — лишняя инфраструктура для одного файла.
- `pipx install` — требует установки `pipx`, что само по себе шаг.
- shell-скрипт с `curl` к API — нет нормальной обработки JSON и tool use.

**Компромисс:**
При росте кода >1000 строк стоит пересмотреть в пользу пакета.

## ADR-002: Единственная внешняя зависимость — `anthropic`

**Статус:** принято

**Контекст:**
Каждая дополнительная зависимость — это потенциальный конфликт версий, отсутствие на машине, необходимость venv.

**Решение:**
Только `anthropic`. Всё остальное — стандартная библиотека Python (`pathlib`, `subprocess`, `argparse`, `getpass`, `json`, `stat`).

**Причины:**
- `anthropic` устанавливается автоматически при первом запуске через `pip install`.
- Стандартная библиотека гарантированно присутствует с Python 3.8+.
- Нет `rich` для красивого вывода — вместо него 30 строк ANSI-кодов вручную.
- Нет `click` для CLI — вместо него `argparse` из stdlib.

**Отвергнутые альтернативы:**
- `rich` — красиво, но лишняя зависимость ради подсветки кода.
- `httpx` напрямую — пришлось бы реализовывать tool use вручную.
- `typer` / `click` — избыточно для 4 аргументов.

## ADR-003: Контекст файловой системы передаётся через `system`, не `user`

**Статус:** принято

**Контекст:**
Модель должна знать содержимое текущего каталога при каждом запросе.
Вопрос: куда именно это передавать в структуре сообщений API?

**Решение:**
Контекст (листинг cwd, системная информация) добавляется к системному промпту, а не к первому `user`-сообщению.

**Причины:**
- `system` сообщение не вытесняется из контекстного окна при длинных диалогах — оно всегда присутствует.
- Семантически верно: контекст окружения — это часть "правил мира", а не вопрос пользователя.
- Позволяет `user`-сообщениям оставаться чистыми (только вопрос).

**Отвергнутые альтернативы:**
- Добавить листинг к первому `user` сообщению — при длинном диалоге вытеснится из окна.
- Передавать каждый раз заново в каждое сообщение — дублирование токенов, дороже.

## ADR-004: История диалога — RAM по умолчанию, опциональная персистентность через `--session`

**Статус:** обновлено

**Контекст:**
В интерактивном режиме нужна память — чтобы второй вопрос "а какой из них самый большой?" понимался в контексте предыдущего ответа.

**Решение:**
По умолчанию `messages: list` живёт в памяти текущего процесса. При явном указании `-s NAME` / `--session NAME` — история сохраняется в `~/.config/ai/sessions/{name}.json` и загружается при следующем запуске.

**Причины:**
- Без флага — поведение не меняется, нет ничего на диске кроме api_key.
- С флагом — контекст накапливается между запусками: `ai -s deploy "что в логах?"` → `ai -s deploy "а вчера?"`.
- JSON-файлы легко инспектировать (`cat`), удалять (`ai --clear-session NAME`), перечислять (`ai --sessions`).
- Сериализация ContentBlock-объектов SDK через `model_dump()` гарантирует совместимость при загрузке.

**Отвергнутые альтернативы:**
- SQLite в `~/.config/ai/history.db` — лишняя сложность для именованных сессий.
- Автоматическое сохранение всех диалогов — вопросы приватности, засорение диска.
- `~/.bash_history`-подобный append-файл — только строки, не пары запрос/ответ.

## ADR-005: `run_command` выполняет произвольные команды без whitelist

**Статус:** принято с оговорками

**Контекст:**
Модель может вызвать `run_command` с любой shell-командой.
Нужно решить: ограничивать ли набор допустимых команд?

**Решение:**
Нет whitelist. Ограничение только на уровне описания инструмента в промпте: "используй только для чтения данных".

**Причины:**
- Whitelist легко обходится (`df` → `df; rm -rf`). Ложная безопасность.
- Настоящая граница безопасности — это права Unix-пользователя. `ai` работает от того же пользователя, что и терминал.
- Пользователь сам решает что делать с предложенными командами — `ai` не выполняет команды **вместо** пользователя, а предлагает их.
- Реальный threat model: модель плохо "галлюцинирует" деструктивные команды при правильном системном промпте.

**Оговорки:**
- `shell=True` в `subprocess.run` потенциально уязвим к инъекции через имена файлов с пробелами/спецсимволами. Принято как допустимый риск в рамках личного инструмента.
- Таймаут 10 секунд предотвращает зависание на `tail -f` и подобных.

**Отвергнутые альтернативы:**
- Whitelist `["df", "ps", "uname", "ls", ...]` — неполный, мешает легитимным задачам.
- `shell=False` с shlex.split — ломает пайпы (`ps aux | grep nginx`).
- Показывать команду и ждать подтверждения пользователя — меняет UX, замедляет работу.

## ADR-006: Лимит `read_file` — 32 КБ

**Статус:** принято

**Контекст:**
Файлы могут быть произвольного размера. Нужен лимит чтобы не переполнить контекстное окно.

**Решение:**
32 768 байт (32 КБ) — жёсткий лимит с уведомлением об обрезке.

**Причины:**
- 32 КБ ≈ 8 000 токенов — значимая часть контекста, но не весь.
- Большинство конфигов, скриптов и логов (хвост) умещаются в 32 КБ.
- При обрезке выводится явное сообщение — пользователь и модель знают что файл неполный.
- Читается через `open("rb")` — явный контроль над количеством байт.

**Отвергнутые альтернативы:**
- 4 КБ — мало для реальных конфигов.
- 256 КБ — рискует переполнить контекст при нескольких файлах.
- Без лимита — опасно для больших логов (гигабайты).
- Стриминг с разбивкой на чанки — усложняет код, модель теряет целостность файла.

## ADR-007: Лимит `list_directory` — 200 записей

**Статус:** принято

**Контекст:**
Каталоги типа `/usr/bin` содержат тысячи файлов.

**Решение:**
200 записей максимум. Если больше — добавляется строка `"... и ещё N объектов"`.

**Причины:**
- 200 имён файлов ≈ 2–4 КБ текста — разумный размер для контекста.
- Для навигации по проекту 200 файлов более чем достаточно.
- При превышении модель видит что список неполный и может уточнить через подкаталоги.

## ADR-008: Автоустановка `anthropic` при первом запуске

**Статус:** принято

**Контекст:**
`anthropic` не входит в стандартную библиотеку. На свежей машине его нет.

**Решение:**
`ensure_deps()` вызывается в самом начале скрипта. Если `import anthropic` падает — вызывается `pip install anthropic` через `subprocess`.

**Причины:**
- Пользователь не должен знать о зависимостях — это детали реализации.
- Аналог поведения `brew` или `cargo` — при первом использовании скачивают что нужно.
- Установка в `--user` scope по умолчанию — не требует sudo.

**Отвергнутые альтернативы:**
- Требовать ручной `pip install anthropic` — лишний шаг, ломает "одна команда для установки".
- `venv` внутри `~/.config/ai/` — усложняет скрипт, нужно активировать окружение.
- Вшить API-клиент через `httpx` напрямую — нужно реализовывать весь протокол tool use вручную.

## ADR-009: Системный промпт заменяется целиком через `system_prompt` файл

**Статус:** принято

**Контекст:**
Пользователи могут захотеть изменить поведение модели под свои нужды (язык, стиль, домен).

**Решение:**
Файл `~/.config/ai/system_prompt` полностью заменяет `DEFAULT_SYSTEM`. Контекст файловой системы добавляется поверх в любом случае.

**Причины:**
- Простая семантика: либо дефолт, либо ваш промпт — без магии мержа.
- Пользователь точно знает что получит — никаких неожиданных остатков дефолтного промпта.

**Отвергнутые альтернативы:**
- `system_prompt` как дополнение к дефолту — неочевидно что войдёт в итоговый промпт.
- Несколько файлов (`prepend`, `append`, `replace`) — избыточно.
- Переменные-плейсхолдеры в промпте — усложняет шаблонизацию.

**Известный недостаток:**
Если хочется добавить только одно правило к дефолту — нужно скопировать весь `DEFAULT_SYSTEM` в файл и дописать. Это неудобно. Возможное решение в будущем: отдельный файл `system_prompt_append`.

## ADR-010: Ввод в интерактивном режиме через `sys.stdin.buffer`

**Статус:** принято

**Контекст:**
`input()` в Python использует кодировку `sys.stdin.encoding`, которая на некоторых Linux-системах (особенно минимальных серверных дистрибутивах через SSH) может быть не UTF-8. При вводе кириллицы возникает `UnicodeDecodeError`.

**Решение:**
Чтение через `sys.stdin.buffer.readline()` с явным `.decode("utf-8", errors="replace")`.

**Причины:**
- Явное лучше неявного: кодировка задана в коде, а не зависит от локали системы.
- `errors="replace"` — вместо краша заменяет некорректные байты на `?`.
- Работает одинаково на системах с `LANG=C`, `LANG=ru_RU.UTF-8` и без локали вообще.

**Отвергнутые альтернативы:**
- `input()` — ломается при не-UTF-8 локали.
- `sys.stdin.reconfigure(encoding="utf-8")` — доступно только в Python 3.7+, но ведёт себя непредсказуемо в некоторых окружениях.
- Принудительный `PYTHONIOENCODING=utf-8` через `install.sh` — патчит окружение снаружи, неочевидно.
